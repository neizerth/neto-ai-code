## Что покажем в скринкасте

- Как сформулировать запрос к ИИ для ревью, чтобы он проверял именно то, что важно команде
- Как разобрать ответ ИИ и быстро отделить полезные замечания от шумных и спорных
- Как оформить итог ревью в PR:
  - комментарии с понятным действием
  - приоритеты (`blocker / major / minor`) и критерии, по которым мы их ставим
  - рекомендации и next steps, которые можно сразу брать в работу

---

## Важная идея занятия

ИИ не заменяет ревьюера.  
ИИ делает «нулевой проход»:

- снимает рутину (формат, очевидные огрехи, повторяющиеся замечания)
- подсвечивает риски, которые легко пропустить при беглом просмотре
- ускоряет старт ревью: ревьюер быстрее понимает, куда смотреть в первую очередь

Финальное решение остаётся за человеком.

---

## План демонстрации

1. Подготовим контекст для ИИ
2. Сформируем промпт на ревью
3. Получим ответ ИИ
4. Классифицируем замечания по «светофору»
5. Оформим ревью в PR
6. Сформируем итоговые рекомендации

---

## Блок 1. Контекст перед ревью

Чтобы ИИ был полезен, ему нужны:

- diff (что изменилось и в каких файлах)
- цель задачи (что должны исправить/улучшить в бизнес-логике)
- критерии качества (что считаем хорошим результатом в нашей команде)
- ограничения (что нельзя сломать: API-контракт, совместимость, безопасность)

---

## Контекст: что считаем качеством

- Корректность и логика
- Читаемость и поддерживаемость
- Безопасность
- Тестируемость
- Соответствие стандартам команды

---

## Блок 2. Промпт для ИИ-ревью

Шаблон запроса:

- Роль ИИ (ревьюер с фокусом на конкретные критерии)
- Контекст проекта (стек, ограничения, цель изменения)
- Что проверить (чек-лист по качеству и рискам)
- Формат ответа (структура и краткость комментариев)
- Приоритизация замечаний (как разделять `blocker / major / minor`)

## Демо 1 — исходный код (`examples/01-express-auth/src/auth.ts`)

```ts
import type { Request, Response } from "express";
import jwt from "jsonwebtoken";

export async function login(req: Request, res: Response) {
  const { email, password } = req.body;
  const user = await findUserByEmail(email);

  if (!user || !checkPassword(password, user.passwordHash)) {
    return res.status(401).json({ error: "Invalid credentials" });
  }

  const token = jwt.sign({ sub: user.id }, process.env.JWT_SECRET!, {
    expiresIn: "1h",
  });

  // Проблема: токен попадает в логи
  console.log("[auth] login success", { userId: user.id, token });

  return res.json({ token });
}
```

## Демо 1 — что ожидаем в ответе ИИ

- `blocker`: «Логирование токена в `console.log`»;
- пояснение риска: токены могут утечь в централизованные логи;
- минимальный фикс: удалить токен из логов или маскировать.

**Переход:** это хороший пример `Accept` (зелёное замечание).

## Блок 3. Ответ ИИ: что делать дальше

Типы замечаний ИИ:

- полезные и точные (можно брать в работу сразу)
- спорные/субъективные (требуют ручной проверки и обсуждения)
- ошибочные (галлюцинации, выдуманные API, неверные предположения)

Наша задача: не принимать всё автоматически.

## Демо 2 — исходный код (`examples/02-tsx-users/src/UserList.tsx`)

```tsx
import { useEffect, useState } from "react";

type User = { id: string; name: string };

export function UserList() {
  const [users, setUsers] = useState<User[]>([]);
  const [query, setQuery] = useState("");

  useEffect(() => {
    fetch(`/api/users?q=${encodeURIComponent(query)}`)
      .then((r) => r.json())
      .then((data) => setUsers(data.items ?? []));
  }, [query]);

  return (
    <>
      <input value={query} onChange={(e) => setQuery(e.target.value)} />
      <ul>{users.map((u) => <li key={u.id}>{u.name}</li>)}</ul>
    </>
  );
}
```

- Компонент списка пользователей.

## Демо 2 — что ожидаем в ответе ИИ

- `major`: нет отмены предыдущего запроса (`AbortController`) — риск гонок;
- `major`: нет обработки сетевой ошибки;
- возможно `minor`: предложение по оптимизации рендера.

**Переход:** это пример `Verify` (жёлтое замечание): сначала проверяем влияние, потом решаем.

---

## «Светофор» для замечаний

- 🟢 **Accept** — исправляем сразу  
  (опечатки, явные баги, очевидные риски)
- 🟡 **Verify** — проверяем вручную  
  (сложный рефакторинг, алгоритмы, производительность)
- 🔴 **Reject** — отклоняем  
  (несуществующие API, вредные рекомендации, галлюцинации)

---

## Пример: зелёное замечание

**Контекст:** в PR есть изменения в `auth.ts`, добавлено новое логирование при ошибке авторизации.  
**Комментарий ИИ:** «Логируется токен доступа в `auth.ts` — риск утечки».

Почему `Accept`:

- факт проверяется по коду за 10 секунд;
- риск высокий: токен может попасть в логи, мониторинг, внешние системы;
- исправление короткое и безопасное (замаскировать/удалить значение).

Что делаем сразу:

- меняем лог на безопасный формат (без токена);
- добавляем короткий комментарий в PR: «исправлено, секреты не логируются».

---

## Пример: жёлтое замечание

**Контекст:** в PR есть поиск по коллекции заказов.  
**Комментарий ИИ:** «Замените текущий алгоритм на двоичный поиск».

Почему `Verify`:

- зависит от структуры данных: двоичный поиск работает только на отсортированных данных;
- может изменить поведение (порядок, обработка дублей, граничные случаи);
- рекомендация может быть полезной, но требует проверки тестами и на реальных данных.

Что делаем:

- проверяем, отсортированы ли данные всегда, а не «в текущем примере»;
- оцениваем выигрыш на реальных объёмах;
- если меняем алгоритм — добавляем тесты на границы и регрессии.

---

## Пример: красное замечание

**Контекст:** PR про авторизацию, слой доступа к данным уже реализован.  
**Комментарий ИИ:** «Используйте метод `findByEmailOrThrow()`».

Почему `Reject`:

- такого метода нет в кодовой базе и в используемой версии библиотеки;
- рекомендация основана на выдуманном API (галлюцинация);
- прямое применение комментария сломает сборку.

Вывод: идею можно сохранить, реализацию — через реальные методы.

Что делаем:

- отклоняем замечание как некорректное;
- при необходимости берём идею (явно обрабатывать «пользователь не найден») и реализуем через существующий API проекта.

## Демо 3 (Node.js / Express): оформление комментария в PR

**Когда показываем:** перед блоком «Оформляем ревью в PR».  
**Цель демо:** показать, как из ответа ИИ сделать рабочий комментарий для команды.
**Файлы демо:**  
`05-review/examples/03-express-orders/src/orders.ts`  
`05-review/examples/03-express-orders/prompt.txt`

---

## Демо 3 — исходный код (`examples/03-express-orders/src/orders.ts`)

```ts
import { Router } from "express";
import { db } from "../db";

const router = Router();

router.get("/orders", async (req, res) => {
  const status = String(req.query.status || "");
  const sql = `SELECT id, total, status FROM orders WHERE status = '${status}'`;
  const rows = await db.query(sql);
  res.json({ items: rows });
});

export default router;
```

---

## Демо 3 — промпт для ИИ-ревью

Берём текст из: `examples/03-express-orders/prompt.txt`

```text

Проверь фрагмент кода маршрута /orders.
Важно:
- найти уязвимости (особенно SQL injection),
- оценить приоритет (blocker/major/minor),
- предложить минимально безопасное исправление.

Формат ответа:
1) проблема,
2) риск,
3) конкретный фикс,
4) приоритет.
```

---

## Демо 3 — как оформляем комментарий в PR

**Проблема:** запрос строится конкатенацией строки с `status`.  
**Риск:** SQL injection при передаче специально сформированного `status`.  
**Фикс:** параметризованный запрос (`WHERE status = ?` / `$1`) через драйвер/ORM.  
**Приоритет:** `blocker`.

**Переход:** именно так превращаем ответ ИИ в рабочий комментарий для команды.

---

## Блок 4. Оформляем ревью в PR

Структура комментариев:

- что не так (коротко и по факту)
- риск / влияние (что может сломаться и где это проявится)
- как исправить (конкретное действие, без размытых формулировок)
- приоритет (зачем исправлять сейчас или можно перенести)

Правило: меньше вкусовщины, больше проверяемых фактов.

---

## Приоритизация замечаний

### Blocker
- уязвимость или утечка данных
- поломка ключевой бизнес-логики
- высокий риск инцидента в проде

### Major
- серьёзное ухудшение поддерживаемости
- заметный риск регрессий или роста техдолга

### Minor
- стиль и локальные улучшения
- рекомендации без критичного риска для релиза

---

## Блок 5. Рекомендации и next steps

После ревью фиксируем:

- что исправить сейчас до merge
- что вынести в отдельную задачу (с приоритетом и владельцем)
- какие тесты добавить, чтобы закрыть риск повторения
- что обсудить на ретро команды, чтобы улучшить процесс ревью

---

## Мини-чеклист перед «Approve»

- Критичные риски закрыты
- Нет утечек секретов
- Поведение не сломано
- Тесты актуальны
- Комментарии по PR закрыты или обоснованно отклонены

---

## Ограничения ИИ (важно проговорить)

- ИИ может уверенно ошибаться
- ИИ не знает весь бизнес-контекст
- ИИ не несёт ответственность за merge

Финальный валидатор — разработчик и ревьюер.

---

## Как внедрять в команде

1. Пилот на одном проекте
2. «Теневой» режим (без блокировки merge, только комментарии)
3. Сбор метрик: приняли / отклонили / ложные + время на разбор замечаний
4. Настройка правил и порогов по итогам первых итераций
5. Расширение на другие репозитории после калибровки

---

## Метрики эффективности

- Время ревью PR
- Доля ложных срабатываний
- Доля принятых рекомендаций
- Количество багов после релиза
- Время на повторные итерации после ревью

Цель: меньше шума, больше пользы.

---

## Итоги

- ИИ в ревью — ускоритель, не автопилот
- Лучший результат: **ИИ + человек**
- Качество зависит от:
  - контекста
  - структуры промпта
  - дисциплины в принятии замечаний
