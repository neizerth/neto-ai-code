**Время прочтения: ~25–30 минут**

**Учебные цели**

- выбираете подходящий под задачу;
- понимаете базовую структуру интерфейса вокруг LLM: ввод → обработка → вывод.

## Введение: как перейти от чата с ИИ к интерфейсу

До этого вы работали с агентом разработки GigaCode: ставили задачу текстом, получали код, уточняли, просили исправить. Это отличный старт — он учит формулировать требования и быстро получать черновик решения.

Но у такого формата есть предел. Как только вы хотите показать решение другому человеку (коллеге, заказчику, пользователю) или проверить реальный сценарий, одного чата становится мало:

- неудобно каждый раз объяснять «что вводить и куда смотреть»;
- сложно воспроизвести один и тот же сценарий несколько раз;
- непонятно, что делать при ошибке (нет сети, лимиты, неправильный ключ);
- трудно собирать обратную связь и логи.

Вот здесь и появляются **быстрые интерфейсы**: небольшой экран с полями ввода, кнопкой запуска и результатом. Это «упаковка» вашей LLM‑логики в форму, с которой может работать не только разработчик.

Представьте: **чат — это разговор на кухне**, а интерфейс — **рецепт с понятными шагами**. В чате вы каждый раз объясняете заново. В интерфейсе шаги уже разложены: «вставь текст → нажми кнопку → получи результат».

Ещё один простой пример из жизни. В чате вы пишете:

- «Сгенерируй письмо клиенту по этому тексту…»

А в интерфейсе это превращается в понятную форму:

- поле «Текст клиента»
- поле «Тон: нейтрально/дружелюбно»
- кнопка «Сгенерировать»
- блок «Результат»

Пользователь заполняет поля и нажимает кнопку.

Когда мы «подключаем нейросеть» к задаче, сначала всё выглядит просто: отправили текст — получили ответ. Но как только появляется пользовательский сценарий, всплывают вопросы:

- где вводить данные (текст, файл, параметры);
- как показать результат на одном экране;
- что делать, если модель отвечает долго или с ошибкой;
- куда записывать логи и как не утечь ключами.

Интерфейс в таких задачах — это способ проверить идею и получить обратную связь без «а давай ещё кнопку» и «а почему не работает».

Почти всегда интерфейс к LLM нужен в одном из трёх случаев:

1) **Демо** — показать идею заказчику или команде.  
2) **Внутренняя утилита** — автоматизировать кусок рутины (например, генерацию черновика письма, разбор логов, первичный анализ текста).  
3) **Проверка пользовательского сценария** — посмотреть, как человек реально пользуется решением: где путается, что вводит, где ошибки.

Интерфейс помогает быстро ответить на главный вопрос: «Это вообще полезно?»

### Сценарий, на котором будем учиться

Собираем **интерфейс “генератор ответа клиенту”**.

Вход:
- текст письма/сообщения клиента;
- тон ответа: нейтрально или дружелюбно.

Выход:
- черновик ответа, который можно отредактировать и отправить.

Этот же сценарий будет появляться в примерах UI, в обработчике и в обработке ошибок.

## №1. Глоссарий

- **UI (пользовательский интерфейс)** — «витрина и кнопки» вашего решения: где человек вводит данные и видит результат.
- **LLM** — большая языковая модель: отвечает в основном текстом, иногда кодом или структурой (например, JSON).
- **API** — «розетка» для программы: способ вызвать модель по сети и получить ответ.
- **Запрос к модели (промпт)** — то, что вы отправляете LLM: инструкции + контекст + входные данные.
- **Логирование** — запись событий: что ввёл пользователь, сколько длился запрос, что пошло не так.
- **Секреты** — ключи доступа (API-ключи, токены). Их нельзя хранить прямо в коде и нельзя случайно показать в интерфейсе/логах.
- **Вредная подсказка (prompt injection)** — попытка через ввод заставить модель нарушить правила: раскрыть секреты, игнорировать инструкции, выполнить нежелательное действие.

## №2. Инструменты для быстрых интерфейсов: Streamlit и Gradio

Ниже — два популярных инструмента для LLM‑демо:

- **Streamlit** — «мини‑приложения» и панели с данными.  
- **Gradio** — формы «ввод → функция → вывод» для демонстраций.

Оба инструмента позволяют за вечер собрать интерфейс, на который в обычной веб‑разработке ушли бы дни.

### 2.1. Как устроена «модель интерфейса» в Streamlit

Streamlit мыслит страницей, на которой код выполняется сверху вниз. Вы пишете:

1) элементы ввода (поле, кнопка, загрузка файла),  
2) обработку (что делаем с введёнными данными),  
3) элементы вывода (текст, таблица, график).

Справочники:
- быстрый обзор возможностей: [Streamlit — Quick reference](https://docs.streamlit.io/develop/quick-reference)  
- полный список компонентов: [Streamlit — API reference](https://docs.streamlit.io/develop/api-reference)

Если упростить, Streamlit — это «скрипт, который рисует страницу».

Пример (каркас «ввод → кнопка → вывод»):

```python
import streamlit as st

st.title("Мини-интерфейс к LLM")

text = st.text_area("Текст для обработки")
style = st.selectbox("Стиль ответа", ["Коротко", "Подробно"])

if st.button("Сгенерировать"):
    # Здесь будет вызов LLM API
    result = f"({style}) Ответ на: {text[:30]}..."
    st.write(result)
```

### 2.2. Как устроена «модель интерфейса» в Gradio

Gradio ближе к идее «есть функция — делаем к ней форму». Вы описываете:

- какие входы (текст, файл, чекбокс),
- какую функцию вызываем,
- какие выходы показываем.

Если нужен сложный экран, используется режим **Blocks**: он даёт макет (строки/колонки/вкладки) и события (нажатие, изменение, отправка).

Полезные страницы:
- основа: [Gradio — Blocks](https://gradio.app/docs/gradio/blocks)  
- события: [Blocks and event listeners](https://gradio.app/main/guides/blocks-and-event-listeners)  
- компоновка: [Controlling layout](https://gradio.app/main/guides/controlling-layout)

Если упростить, Gradio — это «форма вокруг функции».

Пример (тот же каркас, но через «вводы → функция → выводы»):

```python
import gradio as gr

def answer(text: str, style: str) -> str:
    # Здесь будет вызов LLM API
    return f"({style}) Ответ на: {text[:30]}..."

demo = gr.Interface(
    fn=answer,
    inputs=[gr.Textbox(lines=4, label="Текст"), gr.Dropdown(["Коротко", "Подробно"], label="Стиль")],
    outputs=gr.Textbox(label="Результат"),
)

demo.launch()
```

### 2.3. Сравнение Streamlit и Gradio

Практическая разница.

- **“Экран‑панель”:** много блоков, пояснения, несколько шагов, таблицы/графики → Streamlit.  
- **“Демо‑виджет”:** один сценарий, один вызов модели, ввод/вывод → Gradio.

Мини‑таблица для выбора:

| Критерий | Streamlit | Gradio |
| --- | --- | --- |
| Подходит для | панели / утилиты | демо / форма |
| Сложность | средняя | низкая |
| Гибкость UI | выше | ниже |
| Скорость старта | высокая | очень высокая |

**Что выбрать:**

- нужна **одна кнопка и результат** → Gradio
- нужны **несколько блоков и пояснения** → Streamlit

Дополнительно:  
[Streamlit vs Gradio](https://www.deeplearningnerds.com/streamlit-vs-gradio-choosing-the-right-tool-for-building-generative-ai-apps/)

### 2.4. Живые сценарии: какой интерфейс для какой задачи

Три сценария и выбор инструмента.

**Сценарий A: «Покажи демо на созвоне» (5–10 минут).**  
Нужно: один простой экран.

- **Gradio**: форма вокруг одной функции;
- заранее продумать, что показываем при ошибке (иначе демо заканчивается фразой «иногда падает»).

**Сценарий B: «Внутренняя утилита для команды» (используют каждый день).**  
Нужно: несколько шагов, поясняющий текст, настройки, история результатов.

- **Streamlit**: экран‑панель со шагами и пояснениями;
- хранить ключи отдельно, логировать события, ограничить доступ.

**Сценарий C: «Проверка пользовательского сценария» (как поведёт себя человек).**  
Нужно: минимальный набор полей и подсказки.

- подходят оба инструмента, но выбор решает «как проще сделать подсказки и шаги».

Gradio — это «одна кнопка на пульте», а Streamlit — «панель управления с подсказками».

### 2.5. Где чаще всего «ломается ожидание» у новичков

Новичок думает: «Сделаю интерфейс — и всё станет проще». Но интерфейс — это отдельная программа, и у неё есть риски.

Самые частые сюрпризы:

- **Состояние**: пользователь нажал кнопку два раза, а результат «перемешался».
- **Долгие ответы**: пользователь решит, что приложение зависло.
- **Ошибки сети**: нужен текст «что случилось» и «что делать».
- **Безопасность**: демо «на минутку» неожиданно становится доступно всем.

Короткий контраст (как выглядит интерфейс глазами пользователя).

**Плохой интерфейс (типичный):**
- 5 полей ввода без пояснений;
- кнопка `Run`;
- при ошибке ничего не происходит.

**Хороший интерфейс:**
- 1 поле + 1 настройка;
- кнопка «Сгенерировать ответ»;
- статус «Генерируем…» / «Готово» / «Ошибка: попробуйте позже».

Ниже — каркас интерфейса, чтобы эти сюрпризы не застали врасплох.

## №2.6. Как установить Streamlit и Gradio и запустить первый интерфейс

Самый простой путь для старта. **Python уже установлен**, пакеты ставим через `pip` (менеджер пакетов Python).

### 2.6.1. Шаг 0: подготовьте “коробочку” для проекта

Чтобы пакеты не смешивались между проектами, сделайте **виртуальное окружение** — отдельную “коробочку” с зависимостями.

Думайте о виртуальном окружении как об **отдельной полке**. На одной полке лежит Streamlit, на другой — что-то ещё. Ничего не мешается.

В папке проекта выполните:

```bash
python -m venv .venv
```

Активируйте окружение:

- macOS / Linux:

```bash
source .venv/bin/activate
```

- Windows (PowerShell):

```bash
.venv\\Scripts\\Activate.ps1
```

Проверка: после активации команда `python` должна указывать на окружение `.venv`.

### 2.6.2. Установка Streamlit

```bash
pip install streamlit
```

Проверка:

```bash
streamlit --version
```

Запуск приложения:

1) создайте файл `app.py`  
2) запустите:

```bash
streamlit run app.py
```

Обычно Streamlit сам откроет браузер. Если нет — он покажет адрес в терминале (например, `http://localhost:8501`).

### 2.6.3. Установка Gradio

```bash
pip install gradio
```

Проверка:

```bash
python -c "import gradio as gr; print(gr.__version__)"
```

Запуск приложения:

1) создайте файл `demo.py`  
2) запустите:

```bash
python demo.py
```

Gradio выведет адрес в терминале (например, `http://127.0.0.1:7860`).

### 2.6.4. Как пользоваться: базовый цикл «поправил → посмотрел»

У обоих инструментов есть один простой рабочий цикл:

1) написали небольшой код интерфейса;  
2) запустили;  
3) проверили сценарий;  
4) поправили;  
5) повторили.

Начинайте с самого маленького сценария (одно поле → одна кнопка → один результат). Так вы проверите идею и не застрянете в настройках.

### 2.6.5. Частые проблемы и быстрые решения

**Проблема 1: “команда не найдена” (`streamlit: command not found`)**  
Чаще всего вы не активировали виртуальное окружение или поставили пакет в другой Python.

Что сделать:
- активировать `.venv`;
- повторить `pip install streamlit`;
- проверить, что `python -V` и `pip -V` относятся к одному окружению.

**Проблема 2: “порт занят”**  
Если `8501` или `7860` заняты, закройте старый процесс или выберите другой порт.

**Проблема 3: “в браузере пусто / долго грузится”**  
Часто причина — долгий запрос к API. На первых шагах используйте заглушку (как в примерах выше), а потом подключайте реальный вызов.

## №3. Критерии выбора инструмента (и как не ошибиться)

Выбор упирается не в «что моднее», а в ограничения и цели.

### 3.0. Граница применимости

Streamlit и Gradio подходят для демо, внутренней утилиты и проверки сценария. Они мешают, когда нужен «настоящий продукт»:

- много пользователей и роли доступа;
- постоянная авторизация, базы данных, сложные интеграции;
- требования к интерфейсу и производительности как у обычного веб‑приложения.

В этот момент проще перейти на полноценный фронтенд/бэкенд, а Streamlit/Gradio оставить для прототипов и внутренних инструментов.

### 3.1. Простота и скорость

Вопросы:

- интерфейс нужен на **1 день** (показать демо) или на **1 месяц** (внутренняя утилита)?
- сценарий один или их несколько?
- интерфейс должен быть красивым или достаточно «чтобы работало»?

Для старта берите то, что можно показать живому человеку сегодня.

Ещё одно практичное правило: **выбирайте инструмент под форму сценария**.

- Один шаг → одна функция → один результат: Gradio.
- Несколько шагов, пояснения, панель: Streamlit.

И не бойтесь менять инструмент позже. Часто путь такой:

1) Gradio для супербыстрого демо,  
2) Streamlit для внутренней утилиты,  
3) полноценный веб‑интерфейс — когда сценарий «созрел» и стал важным продуктом.

### 3.2. Контроль над кодом и поддержка

Интерфейс, сделанный за вечер, часто остаётся жить. Поэтому заранее подумайте:

- где хранится логика (в одном файле или в модулях),
- как добавлять новые шаги,
- как тестировать ключевые функции.

И Streamlit, и Gradio позволяют начать с одного файла, а потом аккуратно разнести логику по модулям.

### 3.3. Безопасность (ключи, доступ, ввод пользователя)

Проверьте три вещи:

- ключи не лежат в коде и открытых конфигах;
- в логи не попадают токены и другие секреты;
- демо не открыто «всем подряд», если внутри есть чувствительные данные.

**API‑ключ — это ключ от квартиры**. Его нельзя приклеивать на дверь (в код), и точно нельзя показывать «на публику» (в логах/интерфейсе).

Мини‑правило для начала: если вы видите строку вроде `sk-...` в коде или логах — это уже инцидент, даже если «мы же только демо».

Полезные материалы:
- как хранить секреты в Streamlit: [Secrets management](https://docs.streamlit.io/develop/concepts/connections/secrets-management)  
- напоминания по безопасности: [Streamlit security reminders](https://docs.streamlit.io/develop/concepts/connections/security-reminders)  
- если вы делитесь приложением Gradio по ссылке — помнить, что это публичный доступ: [Gradio — Sharing your app](https://gradio.app/main/guides/sharing-your-app)

Отдельная тема — «вредные подсказки» (prompt injection):  
это когда пользовательский ввод пытается заставить модель раскрыть секреты или сделать нежелательное действие. Для базового понимания достаточно OWASP‑страниц из глоссария.

### 3.4. Логирование и наблюдаемость (чтобы понимать, что происходит)

Если вы хоть раз сталкивались с «у меня работает, у пользователя — нет», вы понимаете ценность логов.

Что логировать в LLM‑интерфейсе:

- время запроса и ошибки,
- размер входа/выхода (без чувствительных данных),
- выбранные параметры (модель, температура и т.п. — если есть),
- ключевые этапы сценария (нажал кнопку → отправили запрос → получили ответ).

Пример: что именно писать в логи (без секретов).

- «Запрос генерации ответа клиенту, длина текста: 800 символов»
- «Вызов модели: занял 3.2 секунды»
- «Ошибка сети: таймаут 10 секунд»

А вот чего писать нельзя:

- полный текст пользователя, если там могут быть персональные данные;
- токены и ключи;
- «сырой» ответ, если он содержит чувствительную информацию.

Если вы используете цепочки вызовов (несколько шагов вокруг модели), помогают инструменты трассировки:
- пример базового логирования: [LangChain LoggingCallbackHandler](https://api.python.langchain.com/en/latest/langchain/callbacks/langchain.callbacks.tracers.logging.LoggingCallbackHandler.html)  
- наблюдаемость и оценка качества: [LangSmith evaluation](https://docs.langchain.com/langsmith/evaluation)

## №4. Каркас интерфейса вокруг LLM: ввод → обработка → вывод

Интерфейс к LLM обычно укладывается в схему:

1) **Ввод данных**  
2) **Обработчик (логика)**  
3) **Вывод результата**

Обычно добавляются ещё два слоя: **состояние** и **обработка ошибок**. Они описаны дальше, в разделах №5 и №6.

Разберём каждый блок.

### 4.0. Три быстрых UX‑правила для LLM‑интерфейса

1) Всегда показывайте статус (идёт / готово / ошибка).  
2) Не заставляйте пользователя думать, что вводить: дайте пример и подсказку в поле.  
3) Один экран = один сценарий.

### 4.1. Ввод данных

Типовые элементы ввода:

- текст (запрос пользователя),
- параметры (например, «стиль ответа: кратко/подробно»),
- файл (документ, таблица),
- переключатели (режимы работы).

Делайте ввод минимальным. Чем больше полей, тем больше ошибок и вопросов.

Приём: «один обязательный ввод + одна настройка по умолчанию».

Например:

- обязательный: текст;
- настройка: «коротко/подробно» (по умолчанию «коротко»).

Так пользователь может начать за 5 секунд, а «продвинутые» настройки не мешают новичкам.

### 4.2. Обработчик (логика)

Обработчик — это функция, которая:

- берёт ввод,
- готовит запрос к модели,
- вызывает API,
- обрабатывает ответ,
- возвращает результат для вывода.

Держите обработчик отдельно от UI‑кода (хотя бы как отдельную функцию). Так его проще тестировать и переиспользовать.

UI можно представить как «панель управления», а обработчик — как «двигатель». Панель можно поменять (Streamlit ↔ Gradio), а двигатель оставить тем же.

#### Мини‑пример «двигателя» (одна функция для обоих интерфейсов)

```python
def build_prompt(text: str, style: str) -> str:
    style_hint = "коротко" if style == "Коротко" else "подробно"
    return f"Ответь {style_hint}. Текст: {text}"
```

Если вы перейдёте со Streamlit на Gradio, эта функция останется прежней.

### 4.3. Вывод результата

Вывод бывает разным:

- текст (ответ модели),
- структура (таблица, список),
- файл (скачать результат),
- предупреждения и ошибки.

Вывод — это не только ответ, но и «что делать, если не получилось».

Показывайте не только «результат», но и «состояние».

- «Идёт обработка…»
- «Готово за 3.2 сек»
- «Ошибка: попробуйте позже»

Это уменьшает количество повторных нажатий и дублирующих запросов.

## №5. Состояние и потоковый вывод

Эти два пункта влияют на две вещи: понятность для пользователя и расходы на лишние запросы.

### 5.1. Состояние (что пользователь уже сделал)

Состояние — это ответы на вопросы:

- что введено сейчас;
- что было введено до этого;
- какой последний результат показан;
- в каком режиме мы работаем.

В чате состояние держится само (история переписки). В интерфейсе состояние нужно хранить.

Состояние — это «память» вашего интерфейса. Без неё он как рыбка: каждый раз начинает с нуля.

Без состояния пользователь не понимает, что уже произошло.
Пример: нажал кнопку → получил ответ → нажал ещё раз → старый результат пропал.

Где хранить состояние:

- **Streamlit** — в `st.session_state`.
- **Gradio** — через `gr.State`.

Пример для Streamlit (история ответов):

```python
import streamlit as st

if "history" not in st.session_state:
    st.session_state.history = []

st.session_state.history.append("черновик ответа клиенту")
st.write(st.session_state.history)
```

Пример для Gradio (состояние между нажатиями):

```python
import gradio as gr

def handle(text: str, history: list[str]) -> tuple[str, list[str]]:
    result = f"Ответ на: {text[:30]}..."
    history = history + [result]
    return result, history

with gr.Blocks() as demo:
    inp = gr.Textbox()
    out = gr.Textbox()
    state = gr.State([])
    btn = gr.Button("Сгенерировать")
    btn.click(handle, inputs=[inp, state], outputs=[out, state])
```

### 5.2. Потоковый вывод

Иногда ответ модели длинный. Если ждать молча, пользователь решит, что всё зависло. Потоковый вывод — это когда вы показываете ответ по мере генерации (по кусочкам).

Даже если вы не делаете «настоящий поток», можно хотя бы показывать:

- индикатор «идёт обработка»;
- промежуточный текст «получаем ответ…».

Пример для Streamlit (минимум, который можно вставить сразу):

```python
import streamlit as st

if st.button("Сгенерировать"):
    with st.spinner("Генерируем ответ..."):
        result = call_llm(text, style)
    st.write(result)
```

В Gradio похожий эффект дают индикаторы загрузки: пока функция работает, интерфейс показывает состояние «занято».

## №6. Интеграция с API и обработка ошибок

Когда интерфейс зовёт LLM по API, «сломаться» может многое. Нужны две вещи: короткое сообщение пользователю и подробности в логах (без секретов).

### 6.1. Где именно мы вызываем LLM

В обработчике обычно есть один шаг: «вызвать модель». Тут появляются:

- ключ API,
- модель и параметры,
- сетевые запросы,
- лимиты по частоте (rate limits).

Лимиты — это «турникет в метро». Он не говорит «ты плохой», он говорит «подожди очередь».

### 6.1.1. Мини‑функция вызова LLM (её вызывает UI)

Эту функцию вызывает UI после нажатия кнопки. Она принимает значения, которые вы собрали на экране (`text` и `style`), делает вызов модели и возвращает строку для вывода.

Логика такая:
1) проверяет ввод (например, пустой текст);
2) собирает промпт под ваш сценарий «генератор ответа клиенту»;
3) вызывает API и возвращает ответ или сообщение об ошибке.

```python
import os
from typing import Optional

from openai import OpenAI

client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))


def call_llm(text: str, style: str, *, model: str = "gpt-4.1-mini") -> str:
    if not text.strip():
        return "Введите текст."

    style_hint = "коротко" if style == "Коротко" else "подробно"
    prompt = f"Ответь {style_hint}. Текст: {text}"

    try:
        resp = client.responses.create(
            model=model,
            instructions="Пиши по-русски, без воды.",
            input=prompt,
        )
        return resp.output_text or "Пустой ответ модели."
    except Exception:
        return "Не удалось получить ответ. Попробуйте ещё раз позже."
```

Что важно в этой функции:
- ключ берём из переменной окружения, а не из кода;
- пользователю возвращаем одну фразу без трассировок;
- детали пишем в логи (в примере не показано, чтобы не перегружать блок).

Ссылки на документацию:
- Python‑справочник OpenAI: [API reference (Python)](https://developers.openai.com/api/reference/python)
- лимиты: [Rate limits](https://developers.openai.com/api/docs/guides/rate-limits/)
- как переживать лимиты (повторные попытки): [Cookbook — handle rate limits](https://developers.openai.com/cookbook/examples/how_to_handle_rate_limits/)
- типы ошибок: [Error types guidance](https://help.openai.com/en/articles/6897213-openai-library-error-types-guidance)

### 6.2. Где чаще всего падает (и что показывать)

**1) Нет ключа или он неправильный.**  
Покажите: «не настроен доступ».

**2) Нет сети / таймаут.**  
Покажите: «сервис временно недоступен, попробуйте позже».

**3) Лимиты по запросам.**  
Покажите: «слишком много запросов, подождите немного».

**4) Модель вернула неожиданный формат.**  
Покажите: «не удалось обработать ответ». В логи — подробности.

Для сценария «генератор ответа клиенту» это будет выглядеть так:
- «Не удалось сгенерировать ответ клиенту. Попробуйте ещё раз позже».

Пользователь не должен видеть `Traceback` или текст ошибки API.
Пример:
- ❌ `Invalid API key: sk-...`
- ✅ «Не настроен доступ к сервису»

**5) Слишком длинный ввод (контекст не помещается).**  
Покажите: «слишком длинный текст — сократите или загрузите файл». В логи — длину ввода и лимит, если он известен.

Пример обработки ошибок (идея):

```python
def call_llm_safely(prompt: str) -> str:
    try:
        # вызов LLM
        return "OK: ответ модели"
    except Exception as e:
        return "Не удалось получить ответ. Попробуйте ещё раз позже."
```

## №7. Типичные ошибки новичков (и как их избежать)

Короткий список ошибок, которые встречаются чаще всего.

### 7.0. Чек‑лист перед тем, как дать ссылку на демо

Перед публикацией пройдитесь по списку. Если на любой пункт ответ «не уверен» — лучше не публиковать.

- **Доступ**: ссылка не открывается «всем подряд» (есть пароль, VPN, allow‑list, хотя бы базовая авторизация).
- **Секреты**: в коде и логах нет ключей, токенов, паролей. Поиск по репозиторию на `sk-` и `api_key` ничего не находит.
- **Данные**: вы не показываете чужие тексты/файлы другим пользователям (нет общей «истории» на всех).
- **Логи**: вы не пишете в логи полный ввод пользователя и полный ответ модели.
- **Расходы**: есть лимит на частоту запросов и ограничение на длину ввода. Нет «бесконечного цикла» запросов (когда одно действие пользователя запускает цепочку вызовов без стоп‑условия).
- **Повторы**: нет автоматических повторов без лимита (retry без ограничения, который может разогнать расходы).

### 7.1. «Я выложу демо по ссылке, оно же маленькое»

Риск: демо становится доступным всем, и внутри может случайно оказаться:

- ключи (если вы случайно их показали),
- тексты пользователей,
- результаты, которые не должны быть публичными.

Если вы делитесь приложением, задайте себе вопрос: «Я бы показал это незнакомому человеку?»  
Если ответ «нет» — нужна авторизация, ограничение доступа или локальный запуск.

### 7.2. «Я залогирую всё, чтобы потом разобраться»

Риск: в логи уезжают персональные данные и секреты.
Логи часто хранятся долго и разносятся по направлениям (мониторинг, хранилища, сторонние сервисы).

Лучше логировать «скелет» событий, а не всё содержимое.

### 7.3. «Если нажать кнопку два раза — ну и ладно»

Риск: двойной запрос, перемешанные результаты, лишние расходы.  
Решение: блокировать кнопку на время запроса, показывать «идёт обработка».

### 7.4. «Модель всегда отвечает красиво»

Риск: неожиданный формат, пустой ответ, длинная задержка.  
Решение: предусмотреть «план Б»: таймауты, понятные ошибки, повторные попытки.

## №8. Мини-план: как собрать интерфейс за вечер (без героизма)

Если вы делаете первый интерфейс, идите по шагам. Это экономит время.

1) **Определите один сценарий.**  
   Пример: «вставить текст → получить краткое резюме».

2) **Сделайте каркас UI без модели.**  
   Поле ввода, кнопка, поле вывода. Пусть вывод пока будет заглушкой.

3) **Подключите обработчик.**  
   Одна функция, которая принимает вход и возвращает строку результата.

4) **Добавьте обработку ошибок.**  
   Хотя бы: «нет доступа», «нет сети», «слишком много запросов».

5) **Добавьте логи без секретов.**  
   Время запроса, длины входа/выхода, ошибки.

6) **Проверьте сценарий на 3 типах входа.**  
   Нормальный, пустой, “странный” (очень длинный, с символами, с пробелами).

После этого интерфейс уже можно показывать и улучшать.

## Подведём итоги

- Интерфейс к LLM нужен, чтобы проверять идеи и управлять пользовательским сценарием, а не только «красиво показать ответ».
- Streamlit удобен для **панелей и мини‑приложений**, Gradio — для **форм вокруг функции** и быстрых демонстраций.
- Выбираем инструмент по критериям: **скорость, поддержка, безопасность, логирование**.
- Почти любой LLM‑интерфейс укладывается в схему: **ввод → обработчик → вывод**.
- Ошибки неизбежны. Важно заранее продумать: **как обработать**, **что показать**, **что записать в логи**.
