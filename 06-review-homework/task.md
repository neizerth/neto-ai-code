## Практическое задание «Ревью кода и классификация ошибок»

## Информация для координаторов

- Курс: ARZ
- Модуль: aiprod
- Занятие: Нейросеть как ревьюер
- Тип выполнения: с проверкой

---

## Информация для студентов

**Примерное время выполнения задания** — 60–90 минут.

**Выполнив это задание, вы закрепите навыки:**

- формулировать промпт для ИИ-ревью с контекстом, критериями качества и форматом ответа;
- классифицировать замечания ИИ (корректные / сомнительные / ошибочные) и работать по схеме «светофор»;
- оформлять результат ревью в понятные комментарии с приоритетами (`blocker / major / minor`) и next steps.

**Инструменты и материалы, которые пригодятся:**

- любой ИИ-ассистент (чат) для анализа кода;
- материалы лонгрида «ИИ в code review» (рамка ревью, «светофор», работа с контекстом);
- (опционально) чек-лист по безопасности: [OWASP Code Review Guide](https://owasp.org/www-project-code-review-guide/).

---

## Задание

В этом задании вы отработаете процесс «ИИ как первый ревьюер» на трёх мини-примерах.

Вам нужно:

1) для каждого примера составить **промпт** к ИИ-ревью;  
2) получить ответ ИИ и **отфильтровать** замечания;  
3) разнести замечания по категориям:

- **тип замечания:** корректное / сомнительное / ошибочное;
- **«светофор»:** Accept / Verify / Reject;
- **приоритет:** blocker / major / minor;

4) оформить итог ревью: короткие комментарии и список next steps.

**Конечный результат работы** — документ Markdown, где для каждого из 3 примеров есть:

- исходный код (или ссылка на файл),
- ваш промпт,
- ответ ИИ (вставкой),
- таблица классификации замечаний,
- итоговые комментарии ревьюера и next steps.

## Исходные материалы (код для ревью)

Работаем с **Python, Java и Go**. Ниже — 3 коротких примера кода (их можно просто копировать в чат с ИИ).

### Пример 1 (Python): команда из пользовательского ввода (риск безопасности)

**Ситуация:** утилита запускает системную команду. Аргумент приходит от пользователя.

```python
import os

def run_backup(folder: str) -> int:
    # folder приходит извне (например, из формы)
    cmd = f"tar -czf backup.tgz {folder}"
    return os.system(cmd)
```

### Пример 2 (Java): риск NullPointerException и неявная логика ошибок

**Ситуация:** сервис отправляет письмо пользователю. Репозиторий может вернуть `null`.

```java
public class MailService {
  private final UserRepository userRepository;
  private final MailClient mailClient;

  public MailService(UserRepository userRepository, MailClient mailClient) {
    this.userRepository = userRepository;
    this.mailClient = mailClient;
  }

  public void sendWelcomeEmail(Long userId) {
    User user = userRepository.findById(userId);
    mailClient.send(user.getEmail(), "Welcome!");
  }
}
```

### Пример 3 (Go): SQL-инъекция через конкатенацию строки

**Ситуация:** обработчик принимает параметр `status` и собирает SQL строкой.

```go
package handlers

import (
  "database/sql"
  "fmt"
  "net/http"
)

func Orders(db *sql.DB, w http.ResponseWriter, r *http.Request) error {
  status := r.URL.Query().Get("status")
  query := fmt.Sprintf("SELECT id, total FROM orders WHERE status = '%s'", status)
  _, err := db.Query(query)
  return err
}
```

## Инструкция к выполнению

### Шаг 1. Подготовьте промпт (для каждого примера)

Промпт должен содержать:

- **контекст** (что это за файл и что делает);
- **критерии** (что проверяем: корректность, безопасность, читаемость, тестируемость);
- **ограничения** (например, «не предлагай крупный рефакторинг без причины», «не придумывай новые библиотеки»);
- **формат ответа** (blocker/major/minor + что не так + риск + как исправить).

Подсказка: можно начать с базового шаблона и адаптировать под каждый файл.

### Шаг 2. Получите ответ ИИ и выделите замечания

Скопируйте код в чат с ИИ (или приложите файл) и отправьте промпт.  
Сохраните ответ ИИ в вашем документе.

### Шаг 3. Классифицируйте замечания

Для каждого замечания заполните карточку (по одному замечанию на блок):

- Замечание ИИ: …
- Тип: корректное / сомнительное / ошибочное
- «Светофор»: Accept / Verify / Reject
- Приоритет: blocker / major / minor
- Что делаем дальше: …

Подсказка:

- **Accept** — исправление очевидное и безопасное (например, убрать подстановку пользовательского ввода в команду/SQL);
- **Verify** — нужно проверить влияние тестами/запуском/данными (например, менять поведение обработки ошибок, добавлять отмену запросов, менять алгоритм);
- **Reject** — ИИ выдумал API/пакет или совет вредный.

### Шаг 4. Оформите ревью как комментарии

Для каждого примера составьте 3–6 комментариев «как в PR»:

- **что не так** (1–2 предложения),
- **риск** (почему важно),
- **как исправить** (конкретное действие),
- **приоритет**.

### Шаг 5. Сформируйте next steps

В конце каждого примера добавьте список:

- что исправляем до merge;
- что выносим в отдельную задачу;
- какие тесты стоит добавить.

- [ ] Для каждого замечания есть тип (корректное/сомнительное/ошибочное) и «светофор» (Accept/Verify/Reject).

## Критерии оценки домашнего задания

### «Зачёт»

- Работа выполнена по всем 3 примерам.
- В каждом примере есть промпт, ответ ИИ, классификация замечаний и итоговые комментарии.
- Классификация логичная: приоритеты и «светофор» обоснованы.
- В комментариях нет «вкусовщины» без причин: фокус на рисках и проверяемых фактах.

### «На доработку»

- Есть не все 3 примера или отсутствуют части результата (например, нет классификации или нет next steps).
- Промпт слишком общий (не указаны критерии или формат ответа), из-за чего ответ ИИ шумный.
- Замечания приняты/отклонены без объяснения «почему» и без плана проверки для `Verify`.

### «Незачёт»

Ставится в крайнем случае, если вы присылаете пустое или недоработанное задание во второй раз после отправки работы на доработку.
