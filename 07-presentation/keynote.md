## Якорный пример на всю демонстрацию

Делаем прототип интерфейса **«ответ студенту в учебном чате»**.

Зачем делаем:
- хотим проверить гипотезу: LLM ускорит ответы на типовые вопросы и сделает тон ровнее.

Гипотеза (проверяемая):
- время ответа на типовой вопрос снизится хотя бы на 30%;
- коллеги скажут «это можно отправлять после небольшой правки» в 7 из 10 случаев;
- станет меньше уточняющих вопросов из‑за «непонятного ответа».

Ввод:
- вопрос студента
- тон ответа: нейтрально / дружелюбно

Вывод:
- текст ответа (заготовка для отправки)

## План демонстрации

1. Подключение к API (endpoint, формат запросов/ответов, таймауты)
2. Обработка ошибок (4xx/5xx, пустой/некорректный ответ, минимальные повторы запросов)
3. Проверка сценария «ввод → ответ» (данные и отображение)

## Блок 1. Подключение к API

Нам нужны три вещи:

- **endpoint / клиент** (чем именно вызываем модель)
- **формат запроса** (что отправляем модели)
- **формат ответа** (что показываем пользователю)

---

## Где находится вызов модели

В UI:
- берём `text` и `style` из полей
- по кнопке вызываем обработчик

`style` нужен, чтобы управлять тоном ответа без «магии» в тексте: пользователь выбирает вариант в форме, а обработчик добавляет это как явное правило в запрос к модели.

В обработчике:
- собираем промпт
- зовём API
- возвращаем строку для вывода

---

## Демо: мини‑функция вызова LLM (OpenAI API)

```python
import os
from openai import OpenAI


def call_llm(text: str, style: str) -> str:
    if not text.strip():
        return "Введите вопрос студента."

    tone = "нейтрально" if style == "Нейтрально" else "дружелюбно"
    prompt = f"Сгенерируй ответ студенту {tone}. Вопрос студента: {text}"

    client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))
    resp = client.responses.create(
        model="gpt-4.1-mini",
        instructions="Пиши по-русски, без воды.",
        input=prompt,
    )
    return resp.output_text or "Пустой ответ модели."
```

Для демо можно использовать стартовые бесплатные кредиты/триал (если доступны) — тогда оплата на старте не нужна.

---

## Демо блока 1: подключаем обработчик к Gradio

```python
import gradio as gr

def handle(text: str, tone: str) -> str:
    return call_llm(text, tone)

demo = gr.Interface(
    fn=handle,
    inputs=[
        gr.Textbox(lines=6, label="Вопрос студента"),
        gr.Dropdown(["Нейтрально", "Дружелюбно"], label="Тон ответа"),
    ],
    outputs=gr.Textbox(label="Черновик ответа"),
)

demo.launch()
```

Что показываем:
- UI собрал `text` и `tone`, вызвал `call_llm`, показал строку в поле результата.

## Формат запроса и ответа: что важно

Запрос:
- **инструкции** (как писать)
- **контекст сценария** (что это ответ студенту)
- **входные данные** (вопрос + тон)

Ответ:
- UI ожидает **строку** (её проще показать и логировать)

## Таймауты: правило для интерфейса

Если API «думает» долго:
- пользователь не должен ждать молча
- нужен статус «идёт обработка»

Пример для Streamlit:

```python
import streamlit as st

if st.button("Сгенерировать ответ"):
    with st.spinner("Генерируем ответ..."):
        result = call_llm(text, style)
    st.write(result)
```

## Блок 2. Обработка ошибок

Задача ошибок — не «поймать всё».  
Задача — **не показывать внутренности** и **дать следующий шаг**.

---

## Карта ошибок (в интерфейсе)

- **4xx (доступ/настройка)** → «Не настроен доступ» / «Не хватает прав» → **без повторов**, просим проверить ключ/права
- **5xx (сервис упал)** → «Сервис временно недоступен» → **1 повтор с паузой**, иначе завершаем
- **таймаут / сеть** → «Попробуйте ещё раз» → **1 повтор**, иначе завершаем
- **пустой/битый ответ** → «Не удалось обработать ответ» → предлагаем повторить или упростить ввод

---

## Принцип: пользователю — одно предложение

Пользователь не должен видеть `Traceback` или текст ошибки API.

Пример:
- ❌ `Invalid API key: sk-...`
- ✅ «Не настроен доступ к сервису»

Для нашего сценария:
- «Не удалось сгенерировать ответ студенту. Попробуйте ещё раз позже».

---

## Минимальные повторы запроса (без разгона расходов)

Повтор запроса иногда нужен, но:
- **ограничиваем количество попыток**
- **делаем паузу**
- **не повторяем бесконечно**

Мини‑правило: не больше 1–2 повторов на «сетевую» ошибку, не повторять 4xx.

---

## Демо: «безопасная обёртка» вокруг вызова

```python
def call_llm_safely(text: str, style: str) -> str:
    try:
        return call_llm(text, style)
    except Exception:
        return "Не удалось сгенерировать ответ студенту. Попробуйте ещё раз позже."
```

Что логируем отдельно (без секретов):
- тип ошибки
- длительность запроса
- длину ввода

---

## Демо блока 2: как выглядит ошибка в Gradio

Идея простая: обработчик всегда возвращает строку.

```python
def handle(text: str, tone: str) -> str:
    if not text.strip():
        return "Введите вопрос студента."
    return call_llm_safely(text, tone)
```

Что показываем:
- вместо «падения» пользователь видит одну фразу в результате.

## Блок 3. Проверка сценария «ввод → ответ»

Сценарий считается рабочим, если:
- ввод корректно обрабатывается
- пользователь видит статус
- результат отображается ожидаемо

---

## Теория блока 3: что именно проверяем

Мы проверяем не «качество ответа модели», а сценарий:

- что делаем при пустом вводе;
- что делаем при длинном вводе;
- что видит пользователь, пока идёт запрос;
- что остаётся на экране после нажатия кнопки (старый результат не пропадает «в никуда»).

---

## Проверка ввода (3 кейса)

1) нормальный текст  
2) пустой текст  
3) «странный» текст (очень длинный / много пробелов / символы)

Цель: не падать и не тратить деньги «впустую».

---

## Демо блока 3: три ввода на одном экране

Показываем на примере «ответа студенту»:

1) нормальный текст → получаем черновик ответа  
2) пустой текст → видим «Введите вопрос студента» (без вызова API)  
3) длинный текст → видим «слишком длинный текст…» (и предложение сократить)

---

## Демо: минимальная проверка ввода + сохранение результата

```python
import streamlit as st

if "last_result" not in st.session_state:
    st.session_state.last_result = ""

text = st.text_area("Вопрос студента", placeholder="Например: «Не понимаю, чем 4xx отличается от 5xx…»")
style = st.selectbox("Тон ответа", ["Нейтрально", "Дружелюбно"])

if st.button("Сгенерировать ответ"):
    if not text.strip():
        st.error("Введите вопрос студента.")
    elif len(text) > 8000:
        st.error("Слишком длинный текст — сократите или загрузите файл.")
    else:
        with st.spinner("Генерируем ответ..."):
            st.session_state.last_result = call_llm(text, style)

st.write(st.session_state.last_result)
```

Что показываем голосом:
- пустой ввод → нет запроса к API;
- результат сохраняется на экране до следующего успешного запуска.

## Демо блока 3 (Gradio): проверка ввода + состояние

```python
import gradio as gr

def handle(text: str, tone: str, last: str) -> tuple[str, str]:
    if not text.strip():
        return "Введите вопрос студента.", last
    if len(text) > 8000:
        return "Слишком длинный текст — сократите или загрузите файл.", last

    result = call_llm(text, tone)
    return result, result

with gr.Blocks() as demo:
    text = gr.Textbox(lines=6, label="Вопрос студента")
    tone = gr.Dropdown(["Нейтрально", "Дружелюбно"], label="Тон ответа")
    btn = gr.Button("Сгенерировать ответ")
    out = gr.Textbox(label="Черновик ответа")
    last = gr.State("")
    btn.click(handle, inputs=[text, tone, last], outputs=[out, last])

demo.launch()
```

Что показываем:
- пустой/длинный ввод → показываем одну фразу, `last` не затираем;
- успешный ответ → кладём в состояние и показываем в UI.

## Проверка отображения результата

Проверьте, что UI:

- показывает «Генерируем…» во время запроса
- показывает результат целиком
- не стирает прошлый результат без причины

## Быстрый чек‑лист перед показом демо

- нет ключей и токенов в коде и логах
- нет полного текста пользователя и полного ответа модели в логах
- есть лимит на длину ввода и частоту запросов
- нет повторов запроса без ограничения
