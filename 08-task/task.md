## Информация для студентов

**Примерное время выполнения задания** — 60–90 минут.

**Выполнив это задание, вы закрепите навыки:**

- собирать минимальный интерфейс (ввод → обработчик → вывод) для работы с API;
- подключать обработчик к API (endpoint/клиент, формат запроса/ответа, таймауты);
- обрабатывать ошибки без «traceback на экран» (4xx/5xx, таймаут/сеть, пустой/битый ответ);
- проверять пользовательский сценарий на трёх типах ввода (нормальный, пустой, слишком длинный).

**Инструменты и материалы:**

- Python 3.10+
- Streamlit или Gradio (на выбор)
- доступ к **GigaChat API** (учётные данные/токен согласно документации) в переменных окружения
- [Streamlit docs](https://docs.streamlit.io/)
- [Gradio docs](https://www.gradio.app/docs/)
- [Документация GigaChat API](https://developers.sber.ru/docs/ru/gigachat/api/overview)

## Кейс: интерфейс «список покупок по блюду»

**Контекст:**  
Пользователь пишет блюдо. Приложение возвращает список покупок.

**Ввод (UI):**

- что приготовить (текст)
- на сколько человек: 1 / 2 / 4 / 6
- формат: «Список продуктов» / «Список + шаги»

**Вывод (UI):**

- список покупок (текст, списком)

---

## Задание

Соберите минимальный интерфейс (Streamlit или Gradio), который реализует сценарий **«ввод → обработчик → вывод»** для вызова API модели.

**Обязательные требования:**

- проверка ввода (пустой/слишком длинный) до вызова API;
- обработка ошибок API без вывода traceback и без утечки секретов;
- понятные сообщения пользователю (одно короткое предложение);
- базовая проверка сценария на 3 типах ввода.

**Конечный результат** — ссылка на репозиторий или архив с проектом, в котором есть:

1) приложение (`streamlit_app.py` или `gradio_app.py`);
2) модуль обработчика API (например, `llm.py`) — вызов модели, проверки, ошибки;
3) `README.md` с командами запуска и тест‑сценариями проверки.

---

## Инструкция к выполнению

1. **Подготовьте окружение.**
   - Python 3.10+
   - установите зависимости (Streamlit/Gradio + клиент API)
   - настройте переменные окружения для доступа к **GigaChat API** (по документации провайдера)

2. **Соберите каркас интерфейса.**
   - поле ввода блюда/запроса (многострочный текст)
   - выбор количества людей (выпадающий список)
   - выбор формата ответа (выпадающий список)
   - кнопка «Сгенерировать список покупок»
   - поле вывода результата

3. **Реализуйте обработчик вызова API (в отдельном модуле).**
   - вход: `dish: str`, `people: str`, `format: str`
   - выход: строка результата
   - сформируйте запрос к модели так, чтобы количество людей и формат ответа управлялись явно
   - требование к формату ответа: список продуктов (по строке на продукт). Если выбран «Список + шаги», добавьте короткие шаги приготовления (5–8 пунктов).

4. **Добавьте валидацию ввода (до вызова API).**
   - пустой ввод → «Напишите, что хотите приготовить.»
   - слишком длинный ввод (выберите порог) → «Слишком длинный текст — сократите или разбейте на части.»

5. **Добавьте обработку ошибок API.**
   Рекомендованная карта ошибок (минимум):
   - 4xx (доступ/настройка) → «Не настроен доступ к сервису.» (без повторов)
   - 5xx (сервис недоступен) → «Сервис временно недоступен. Попробуйте позже.» (максимум 1 повтор по желанию)
   - таймаут/сеть → «Не удалось получить ответ. Попробуйте ещё раз.» (максимум 1 повтор по желанию)
   - пустой/битый ответ → «Пустой ответ модели. Попробуйте переформулировать вопрос.»

6. **Проверьте сценарий «ввод → ответ» на 3 кейсах.**
   - нормальный текст → получаете список покупок
   - пустой текст → сообщения в UI, вызова API нет
   - длинный текст → сообщения в UI, вызова API нет

## Чек-лист самопроверки результатов

- [ ] UI реализует каркас **ввод → обработчик → вывод** (поле текста, два выпадающих списка, кнопка, поле результата).
- [ ] Вызов API вынесен в отдельный модуль/функцию (не размазан по UI‑коду).
- [ ] Пустой ввод не отправляется в API (есть понятное сообщение).
- [ ] Есть ограничение на длину ввода (есть понятное сообщение).
- [ ] Ошибки API не «падают» traceback’ом в интерфейс.
- [ ] `README.md` содержит: команды запуска, описание 3 проверок сценария, и как воспроизвести хотя бы 1 ошибку (например, убрать ключ).
- [ ] Секреты не пишите в коде. Берите их из переменных окружения.

## Критерии оценки домашнего задания

### «Зачёт»

- Интерфейс запускается и демонстрирует сценарий «ввод → обработчик → вывод».
- Реализованы проверки ввода (пустой/длинный) и обработка ошибок API без traceback.
- В проекте есть `README.md` с запуском и тест‑сценариями проверки.

### «На доработку»

- Приложение запускается, но:
  - нет проверок пустого/длинного ввода, и/или
  - ошибки выводятся в виде traceback/сырых исключений, и/или
  - отсутствует `README.md` или он не позволяет повторить проверки.

### «Незачёт»

Ставится в крайнем случае, если вы присылаете пустое или недоработанное задание во второй раз после отправки работы на доработку.
